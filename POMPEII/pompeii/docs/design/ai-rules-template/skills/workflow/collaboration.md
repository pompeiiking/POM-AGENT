# 协作行为准则

## 核心原则

**你不是项目中唯一的开发者。** 人类开发者、其他AI助手可能同时或先后在同一项目上工作。你的每次修改都会被其他人看到和依赖。

## 规则

### 1. 先读后写

- 修改任何文件前，先阅读 `STATUS.md` 了解项目当前状态
- 修改任何文件前，先读完该文件和它的上下游依赖
- **不理解的代码不要修改、不要删除、不要"优化"**

### 2. 最小变更原则

- 每次修改只做当前任务需要的变更，不顺手重构
- 如果发现代码有改进空间但不在当前任务范围：
  → 记录到 STATUS.md 的"已知问题"中
  → **不要自作主张修改**

### 3. 不引入新依赖（除非明确授权）

- **不允许自行安装新的 npm/pip/cargo 包**
- 需要新依赖时，向人类开发者说明：
  → 为什么需要？现有依赖能否解决？
  → 包的大小、维护状态、安全性
- 人类开发者批准后再安装

### 4. 不删除不理解的代码

- 看起来无用的代码可能是为了特定边界情况
- 不确定时，问而不是删
- 如果确认无用，先标注 `// TODO(日期): 确认后可删除 — 原因`，下次再删

### 5. 锁定区域

遇到以下标记的代码块，**不允许修改**：
```
// @lock — [原因]
/* @lock — [原因] */
# @lock — [原因]
```

### 6. 规范文档修改门槛

修改**规范文档**（具体清单见 changelog skill "规范文档变更的特殊要求"）的门槛高于普通代码，因为影响所有开发者/AI行为：

- **必须有明确授权** — 人类开发者要求修改，或完整性检查发现确实的缺陷（如索引与文件不一致）
- **修改前必须向人类开发者说明** — "我要修改[X文件]的[Y内容]，因为[Z原因]，你确认吗？"
- **一次修改只解决一个问题** — 不要在同一次修改中混合多个不相关的规范变更
- **修改后必须用规范变更模板记录CHANGELOG** — 格式见 changelog skill
- **修改后必须自验** — 具体检查项：
  - RULES.md Skill索引条目 ↔ 实际skill文件是否对应
  - RULES.md 优先级列表(L1/L2/L3) ↔ skill frontmatter priority是否一致
  - skill的depends字段 ↔ 被引用的skill是否存在
  - 任务预设组合中的skill名 ↔ 索引中是否都有
  - 铁律条数描述 ↔ 实际铁律数量是否吻合

✅ 正确做法：
```
AI: "完整性检查发现 RULES.md 优先级列表缺少 interface-change，
      但它在 frontmatter 中标记为 L1。
      我需要在 L1 列表中补上它。你确认吗？"
人类: "确认"
(修改 + 规范变更CHANGELOG)
```

❌ 错误做法：
```
AI: (您没有说什么，直接修改了RULES.md的5条铁律、优先级列表、接入协议，
     没有问确认，没有记录变更前/后对比)
```

### 8. 删除文件/移除功能流程

删除代码比新增代码更危险——别人可能依赖你要删的东西：

- **删除前全局搜索引用** — 搜索文件名、导出的函数/类/类型名，确认无其他文件依赖
- **更新相关 index 文件** — 移除已删除文件的导出
- **更新相关测试** — 删除或调整依赖该文件的测试
- **确认不在契约中被引用** — 如涉及 `contracts/` 中定义的接口，需人类开发者确认
- **不确定能否删 → 先标记再删**：`// TODO(日期): 确认后可删除 — 原因`

## 提交前检查清单

- [ ] 已阅读 STATUS.md 了解项目当前状态
- [ ] 修改范围仅限当前任务，没有超出范围的"顺手重构"
- [ ] 没有安装新的依赖包（或已获得授权）
- [ ] 没有删除不确定用途的代码
- [ ] 没有修改带 `@lock` 标记的代码块
- [ ] 所有变更已记录到 CHANGELOG.md
- [ ] 若修改了规范文档，已获得授权且使用规范变更模板记录
