Pompeii - Agent 通用模块架构设计 ver0.4
=========================================


一、架构总览
-----------

Agent 采用微内核架构。核心部做 Loop 级路由，三个处理模块同级并列，
五个共享服务按需供给。Agent 不关心传输方式，AgentPort 是唯一边界。

┌─ 外部（部署环境）────────────────────────────────┐
│  通信管理 / 频道适配 / 设备管理                    │
└──────────────────────┬───────────────────────────┘
                       │
               ┌───────┴───────┐
               │   AgentPort   │  Agent 唯一边界
               │               │
               │  handle():    │  user_message / device_result / system_command
               │  emit():      │  reply / device_request / status / delegate / confirmation
               └───────┬───────┘
                       │
┌──────────────────────┴───────────────────────────┐
│                                                   │
│  ┌─────────────────────────────────────────┐      │
│  │           核心部（微内核）                │      │
│  │                                         │      │
│  │  · Loop 级模块间路由                     │      │
│  │  · Agent Loop 循环控制 + 终止判断        │      │
│  │  · 工具调用安全检查（关卡③）             │      │
│  │  · 资源写操作守门                        │      │
│  │  · 会话管理（创建/查找/状态切换/归档）    │      │
│  │  · 编排级容错                            │      │
│  │  · 生成 request_id                      │      │
│  └──┬──────────────┬──────────────┬────────┘      │
│     │              │              │                │
│     ▼              ▼              ▼                │
│  组装部          模型部          工具部             │
│  (一切准备)      (一切推理)      (一切执行)         │
│     │     直接调用  │     直接调用  │                │
│     ▼       ↓      ▼       ↓      ▼                │
│  ┌──────────────────────────────────────────┐      │
│  │  共享服务: 提示词配置 / Skill注册表        │      │
│  │  信息缓存 / 长期存储 / 安全策略模块        │      │
│  └──────────────────────────────────────────┘      │
│  ┌──────────────────────────────────────────┐      │
│  │  日志基础设施（独立，各模块直接写入）       │      │
│  └──────────────────────────────────────────┘      │
└──────────────────────────────────────────────────┘

路由判断标准:
  是 Agent Loop 的步骤（组装→推理→工具→...）→ 经核心路由
  是模块内部子步骤（辅助模型调用、资源查询）→ 直接调用共享服务


二、核心运行流程
---------------

① AgentPort 收到消息 → 核心生成 request_id → 会话查找/创建 → 路由到组装部

② 组装部（完整组装，内部自行编排，不经核心）:
   · 预处理用户输入（多模态识别，必要时调模型部辅助模型）
   · 读提示词配置 / Skill注册表 / 信息缓存 / 长期存储
   · 组装 Context Window + Token 预算裁剪
   → 返回核心

③ 核心路由到模型部 → 主推理 → 流式输出 → 输出解析

④ 输出是 tool_call:
   → 核心执行关卡③: 权限/人工确认/沙箱标记/[守卫模型意图审核]
   → 通过 → 核心路由到工具部
   → 工具部执行关卡④: 网络策略/信任分级/沙箱隔离 → 执行 → 结果净化
   → 净化结果经核心写入信息缓存
   → 核心路由到组装部（追加路径）→ 更新 Context Window
   → 回到 ③（循环）

⑤ 输出是文本回复:
   → 核心路由到组装部（格式化） → AgentPort.emit("reply")
   → 写入信息缓存 → 必要时触发会话归档


三、处理模块
-----------

三个处理模块地位相同，由核心部在它们之间路由。

3.1 组装部

  职责: Context Window 的唯一构建者。为模型准备所有输入，处理所有输出。

  完整组装路径（首次）:
    1. 预处理用户输入
       · 可直通模态（文本/图片/音频）→ 保留原始格式
       · 需预处理模态（视频→抽帧、文件→提取内容）→ 并行处理
       · 模型不支持某模态时 → 调模型部辅助模型转换
    2. 读提示词配置 → System Prompt
    3. 读 Skill 注册表 → 工具声明 + Skill 描述
    4. 读信息缓存 → 会话历史
    5. 读长期存储 → 检索相关记忆（用用户原话作为检索 key）
    6. 组装 Context Window（多模态内容块数组）
    7. Token 预算计算 + 裁剪
       文本: tokenizer 计算 / 图片: 按尺寸折算 / 音频: 按时长折算

  追加路径（循环中工具结果回注时）:
    1. 格式化 + 截断工具返回结果
    2. Token 预算检查，超限则压缩（三级策略，见第七节）
    3. 追加到现有 Context Window

  出方向: 格式化模型最终回复（平台适配、分片）

  上下文隔离（关卡②）: 组装时按安全策略模块的隔离规则，
  用结构化标签分离系统区/记忆区/用户区/工具结果区，标记来源和信任等级。

3.2 模型部

  双重角色:

  角色 A — 处理模块（Agent Loop 主推理）:
    · 模型选择（按会话配置）
    · API 调用 + 流式输出
    · 输出解析（tool_call vs 文本）
    · Failover: 主模型失败 → 切备用模型

  角色 B — 共享服务（辅助模型）:
    · model.stt(audio) → text            语音转文字
    · model.describe(image) → text        图片描述
    · model.embed(text) → vector          向量嵌入
    · model.summarize(text) → text        文本压缩
    · model.guard(content, type) → SAFE/UNSAFE   安全审核

  辅助模型 vs Skill:
    辅助模型: 系统内部调用，不经 tool_call，不进 Context Window
    Skill: 模型通过 tool_call 调用，描述文本注入 Context Window

  统一管理:
    · 模型注册表（主推理 + 辅助 + 守卫）
    · API Key 集中管理 / 连接池 / 重试 / Failover
    · 熔断器: 连续失败自动熔断，定期试探恢复
    · 调用计量

  守卫模型（Guard Model）:
    · 独立上下文，不受主模型污染
    · 安全专用提示词，小/快/便宜模型，二分类输出
    · 三种审核: 输入审核(关卡②) / 意图审核(关卡③) / 内容审核(关卡④)

3.3 工具部

  职责: 执行模型发出的所有 tool_call

  三类执行器:
    · 本地执行器: 在 Agent 服务器上执行（文件/Shell 等）
    · 远程执行器: MCP Client，连接外部 MCP Server
    · 设备执行器: 通过 AgentPort 请求用户设备能力（摄像头/麦克风/GPS）
      flow: 工具部→核心→AgentPort.emit("device_request")
            →外部设备返回→AgentPort.handle("device_result")→核心→工具部

  获取数据 vs 处理数据:
    获取（拍照/录音）→ 工具部设备执行器
    处理（理解图片/语音转文字）→ 模型部辅助模型
    两者独立。

  执行安全层（关卡④，详见第六节）:
    网络策略 / 信任分级 / 结果净化 / 沙箱隔离

  Workflow: 多步骤确定性流程封装为 Workflow，作为 Tool 注册到 Skill 注册表。

  内部容错: 重试逻辑 + 熔断器（跟踪 MCP Server / 设备连接健康状态）


四、共享服务
-----------

不参与 Agent Loop 流转，被处理模块按需查询。

4.1 提示词配置
  生命周期: 系统级，基本不变
  内容: System Prompt 模板、人格定义（SOUL.md）、行为规则、输出格式约束
  消费者: 组装部

4.2 Skill 注册表
  生命周期: 系统级，可热更新
  内容: 每个 Skill 打包三部分:
    (1) 描述文本 → 注入 Context Window，让模型知道能力存在
    (2) 可执行函数 → 工具部执行
    (3) 元数据 → 参数 Schema、权限要求、沙箱策略
  消费者: 组装部（描述+Schema）、工具部（函数+配置）
  Skill ≠ Tool: Tool 是原子操作，Skill 是能力包。

4.3 信息缓存
  生命周期: 会话级
  内容: 当前会话 messages 列表（用户消息/模型输出/工具结果）
  消费者: 组装部读，循环引擎每轮写（经核心守门）
  多模态: 会话中保留原始数据，归档时转文本摘要

4.4 长期存储
  生命周期: 持久级，跨会话
  内容: 向量索引 + BM25 索引（混合检索）、对话归档、用户偏好
  检索: 向量相似度 + BM25 混合，用用户原话做查询
  嵌入: 调模型部 model.embed()
  可插拔: 默认 SQLite + sqlite-vec，可替换

4.5 安全策略模块
  定位: 横切面规则库。自身不执行检查，被各关卡查询。
  规则清单:
    关卡① | 输入校验规则: 长度/频率/格式
    关卡② | 上下文隔离规则: 分区标签格式、信任等级标记
    关卡③ | 工具权限矩阵 / 人工确认清单 / 沙箱策略
    关卡④ | 网络访问策略 / 信任等级表 / 净化规则
    关卡⑤ | 资源访问策略: 读写分级规则
    守卫   | 守卫模型触发策略


五、资源访问规则
---------------

规则1: 处理模块间通信（Loop 步骤）   → 经核心路由
规则2: 资源写操作                   → 经核心守门
规则3: 低/中敏感资源读操作           → 直连 + 事件上报
规则4: 高敏感资源读操作              → 经核心审批

敏感度分级:
  低   | 提示词配置、Skill注册表    | 直连+上报     | 静态/半静态
  中   | 信息缓存                  | 直连+详细审计  | 含用户会话
  高   | 长期存储(私密记忆)、凭证    | 经核心审批     | 泄露不可逆


六、安全架构
-----------

五道关卡 + 规则库 + 守卫模型。
规则检查永远执行，守卫模型高风险时触发。信任越低、风险越大，检查越严格。

全链路:
  关卡① → 关卡② → 模型推理 → 关卡③ → 关卡④ → 结果回注
  入口     上下文              工具      执行
  过滤     隔离               安全门     安全层
  ────────────── 关卡⑤ 数据访问控制（贯穿全程）──────────────

6.1 关卡① 入口过滤
  位置: AgentPort / 外部环境
  检查: 消息长度 / 频率 / 身份验证 / 格式校验
  可选: 守卫模型输入审核

6.2 关卡② 上下文隔离
  位置: 组装部
  机制: 用结构化标签将 Context Window 分为系统区/记忆区/用户区/工具结果区
  每区标记来源和信任等级，防止 prompt injection

6.3 关卡③ 工具安全门
  位置: 核心部（tool_call 路由到工具部之前）
  检查顺序:
    A. 权限: 查工具权限矩阵 → 不允许则拒绝
    B. 人工确认: 查确认清单 → 需要则 AgentPort.emit("confirmation")
    C. 沙箱标记: 查沙箱策略 → 标记 sandbox=true/false
    D. [可选] 意图审核: 高危时调守卫模型，检查 tool_call 是否符合用户意图
  原则: 工具部只收到已通过所有检查的调用

6.4 关卡④ 执行安全层
  位置: 工具部内部
  四道子机制:
    ④-a 网络策略: 每个工具的域名白名单/黑名单、禁止下载类型、防SSRF、超时
    ④-b 信任分级:
      完全信任(内置工具): 最少检查
      高信任(官方MCP): 轻量检查
      低信任(第三方MCP): 参数脱敏 + 强制超时 + 完整净化
      不信任(未知来源): 禁止连接
    ④-c 结果净化（回注上下文前）:
      长度截断 → 注入检测(规则匹配) → [守卫模型审核(低信任)] → 来源标签封装 → 可执行内容过滤
    ④-d 沙箱隔离: 独立文件系统 / 网络受策略控制 / 资源限制 / 执行后销毁

  成本控制: 信任等级越高检查越轻，越低越重。

  实现补充（主线代码，与④-c「来源标签封装」对齐）:
    组装部 apply_tool_result 将 tool <name> -> <payload> 形式的回流摘要包入关卡② tool_result 分区
    （与 §6.2 相同 HTML 注释形态）；ToolResult.source（local / mcp / device 等）映射为分区 trust 档
    （内置偏高、外接 MCP/设备偏低）。模型部在 tool_first 等路径解析时会先剥除该外壳再读取内层文本。
  实现补充（与④-c「注入检测(规则匹配)」对齐）:
    核心部写入会话前对工具输出做子串规则扫描；结构化输出按 JSON 文本检视（与组装序列化一致）。
    默认安全策略含对伪造 pompeii 分区注释边界的拦截（见 security_policies.tool_output_injection_patterns）。

6.5 关卡⑤ 数据访问控制
  详见第五节资源访问规则。
  凭证额外保护: 加密存储、权限收窄、排除版本控制。


七、会话生命周期
---------------

7.1 数据模型

  Session
    session_id / user_id / channel / status
    config: { model, skills, security, limits: { max_tokens, max_context_window, max_loops, timeout } }
    stats: { total_tokens_used, message_count, created_at, last_active_at }
    messages[]

  Message
    message_id / role(user|assistant|tool) / parts[] / timestamp / loop_index / token_count

  Part
    type(text|image|audio|tool_call|tool_result) / content
    metadata: { source, trust, token_count }

7.2 状态机

  收到消息(无匹配) → ACTIVE ←── 收到消息(IDLE)
                       │
                 一段时间无消息
                       ▼
                     IDLE
                       │
           归档阈值 / 用户结束 / token耗尽
                       ▼
                   ARCHIVED
                       │
                 删除 / 过期
                       ▼
                   DESTROYED

  ACTIVE:
    · 消息解析为 Message→Part，写入信息缓存
    · 走 Agent Loop，更新统计
    · 监控: 窗口超限→压缩(组装部)，总预算超限→终止(核心部)

  IDLE: 信息缓存保留，停止计时器。用户回来直接恢复。

  ARCHIVED:
    · 对话写入长期存储，多模态→文本摘要
    · 提取关键事实更新长期记忆
    · 释放信息缓存，保留 session 元数据
    · 用户未来回来创建新 session，长期存储自然延续上下文

  DESTROYED: 删除所有数据，不可恢复。

7.3 Token 管理

  层面A — 单次窗口（组装部负责）:
    每次组装时检查总 token，超限触发三级压缩:
      1. 裁剪工具中间结果（tool_call+result→摘要）
      2. 压缩早期对话（调 model.summarize()）
      3. 删除最早消息（最后手段）
    被压缩的原始数据异步归档到长期存储。

  层面B — 会话总消耗（核心部负责）:
    每次模型调用后累加 token。超预算→强制终止 Loop。

7.4 职责归属
  核心部: session 生命（创建/查找/状态切换/超时/归档触发/总预算）
  信息缓存: session 数据（messages）
  长期存储: session 记忆（归档/画像）
  组装部: session 数据使用（组装 Context Window、窗口级压缩）


八、容错与终止
-------------

8.1 循环终止条件
  · 模型输出文本回复（正常结束）
  · 最大循环次数（如 15 轮）
  · Token 预算耗尽
  · 单次请求超时
  · 重复 tool_call 检测

8.2 三层容错
  层次1 — 模块内部自愈（对核心透明）:
    · 模型部: 重试→Failover→熔断
    · 工具部: 重试→熔断
    · 长期存储: 异常→降级返回空结果

  层次2 — 核心编排级:
    · 模型不可用 → AgentPort.emit("reply", "服务暂时不可用")
    · 工具失败 → 错误回注上下文，让模型自行调整
    · 循环超限 → 强制终止

  层次3 — 日志基础设施（独立）:
    · 各模块独立写日志，带 request_id 串联链路


九、多模态处理
-------------

模态融合是模型的工作，架构保持原始格式不过早降维。

输入（组装部内并行处理）:
  文本/图片/音频 → 直通（模型原生支持时）
  音频 → 调 model.stt()（模型不支持时）
  视频 → 抽关键帧为图片序列
  文件 → 提取文本或转图片

Context Window 结构为内容块数组:
  { role, content: [{ type: "text"|"image"|"audio", ... }] }

存储: 信息缓存保留原始数据，长期存储归档为文本摘要+向量嵌入。


十、可扩展性
-----------

可插拔扩展点:
  · Skill 注册表: 新 Skill 即插即用
  · 模型部: 模型提供商接口（Claude/GPT/本地模型）
  · 长期存储: 存储引擎接口（SQLite/PostgreSQL/向量数据库）
  · 长期存储: 检索策略接口（混合检索/GraphRAG）
  · AgentPort: 消息类型可扩展

推理策略可插拔:
  · ReAct（默认）: 通用对话任务
  · Plan-and-Execute: 复杂多步任务
  · Workflow: 确定性流程

多 Agent 协作:
  连接机制已内置（AgentPort 的 delegate）。
  工作策略（层级/流水线/并行/辩论）通过 Prompt + Skill + Workflow 配置实现。
  实现补充（主线）: 用户指令 `/delegate <target> <payload>` 经核心登记会话后由 Port 发出
  `DelegateEvent(target, payload)`；子 Agent 实例化与 RPC 由部署侧网关消费该事件完成。
  可选 `kernel.delegate_target_allowlist` 收紧允许转发的 target；未命中则拒绝且不写会话。
