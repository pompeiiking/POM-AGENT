# Pompeii-Agent 系统测试流程

本文档描述对 Pompeii-Agent 进行端到端系统测试的步骤、用例与通过标准，便于发布前或迭代后做回归验证。会话存储以 **`runtime.yaml` → `backend: sqlite`** 为准（实现细节见 `docs/design/开发状态与系统接口.md` 与 `runtime.yaml`）。

**文档版本**：与 `src/app/version.py` 对齐（当前 **0.4.7**）。

**复制粘贴说明**：代码块中的命令均为单行、无换行续行符，请整行复制到 PowerShell 执行，避免复制到多余空格或换行导致报错。

---

## 一、前置条件

### 1.1 环境与依赖

- Python 3.11+，项目依赖已安装：`pip install -r requirements.txt`
- 代码根目录为 **本仓库根目录**（包含 `src`、`requirements.txt` 的目录）

### 1.2 API Key（模型对话与工具链路依赖）

- 若需验证「普通对话」与「上下文维持」，须配置 DeepSeek API Key（环境变量 `DEEPSEEK_API_KEY`）。
- **推荐**：在仓库根目录创建本地 **`env.ps1`**（已 gitignore），内容为 `$env:DEEPSEEK_API_KEY = "..."`；勿提交。
- 若 PowerShell **禁止执行** `. .\env.ps1`：在同一终端粘贴一行  
  `Get-Content -Path .\env.ps1 -Raw | Invoke-Expression`（见 `docs/guides/API密钥配置操作手册.md`），或直接用 **`.\scripts\run-http.cmd`** 启动服务。
- 详细说明见 `docs/guides/API密钥配置操作手册.md`。

### 1.3 启动 HTTP 服务（自动注入 API Key）

在本仓库根目录下执行（**一条命令**，会先加载 `env.ps1` 再启动服务；无 `env.ps1` 时会提示警告但仍可启动，仅模型用例会失败）。

**若 PowerShell 提示「禁止运行脚本」**（ExecutionPolicy），请用下面 **`.cmd` 方式**（不修改系统策略）：

```
cd C:\Users\22271\Desktop\Agent
.\scripts\run-http.cmd
```

或直接 PowerShell：

```
cd C:\Users\22271\Desktop\Agent; .\scripts\run-http.ps1
```

若仍被拦截，可用（当前会话绕过策略执行该脚本文件）：

```
cd C:\Users\22271\Desktop\Agent
powershell -NoProfile -ExecutionPolicy Bypass -File .\scripts\run-http.ps1
```

等效手动方式（每行单独复制）：

```
cd C:\Users\22271\Desktop\Agent
. .\scripts\load_env.ps1
$env:PYTHONPATH = "src"
python -m app.http_runtime
```

确认输出无报错，服务监听 `http://0.0.0.0:8000`。以下用例均默认使用该 HTTP 接口。

**说明**：`Invoke-RestMethod` 测试命令**不需要**在客户端再设 Key；Key 只需在**运行 `http_runtime` 的进程**中通过上述方式注入。

可选：长期允许本地脚本可执行：`Set-ExecutionPolicy -Scope CurrentUser RemoteSigned`（需自行确认安全策略）。

## 二、测试用例与执行步骤

### 2.1 健康检查

**目的**：确认服务存活、路由可用。

**步骤**（整行复制执行）：

```
Invoke-RestMethod -Uri "http://127.0.0.1:8000/health"
```

**通过标准**：HTTP 200；JSON 含 `ok: true`，且含 **`version`** 字段（与 `src/app/version.py` 一致）。

---

### 2.2 普通对话（单轮）

**目的**：验证 Port → Core → Assembly → Model 链路，模型能返回文本回复。

**步骤**（整行复制到 PowerShell 执行）：

```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http","text":"你好，请用一句话介绍你自己"}'
```

**通过标准**：

- HTTP 200，返回体包含 `events` 数组。
- `events` 中有一条 `kind: "reply"`，且 `text` 为模型生成的自然语言（非“未配置 API Key”等错误提示）。

---

### 2.3 上下文维持（同一会话多轮）

**目的**：验证同一 `user_id` 下多次请求共享会话，模型能基于历史回答“上一句说了什么”。

**步骤**：

1. 第一句（建立上下文），整行复制执行：
   ```
   Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http","text":"我叫小明，喜欢跑步和编程"}'
   ```
2. 第二句（依赖上下文），整行复制执行：
   ```
   Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http","text":"我上句话说的是什么？请简要复述"}'
   ```

**通过标准**：

- 两次请求均返回 `kind: "reply"`。
- 第二次回复中应能体现对第一句内容的引用或概括（如“小明”“跑步”“编程”等），而不是“无法看到历史记录”一类表述。

---

### 2.4 系统指令：/help

**目的**：验证 Port 意图解析与 Model 本地规则分支，不调用外部模型即返回帮助文案。

**步骤**（整行复制执行）：

```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http","text":"/help"}'
```

**通过标准**：

- 返回 `events` 中有一条 `kind: "reply"`。
- `text` 中包含项目约定的帮助说明（如“Pompeii-Agent”“/help”“/summary”“/tool echo”等），且风格为本地固定文案，而非模型自由发挥的通用介绍。

---

### 2.5 系统指令：/summary

**目的**：验证 /summary 意图走本地规则分支并返回基于最近消息的**规则摘要**（不调用外部模型）。

**步骤**（整行复制执行）：

```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http","text":"/summary"}'
```

**通过标准**：

- 返回 `kind: "reply"`，正文以「最近对话概要（规则生成，非模型）」开头，并列出最近若干条 `[用户]/[助手]` 等摘要行；无历史时可提示「暂无对话记录」。

---

### 2.6 工具调用与确认：/tool echo（需确认时）

**目的**：验证工具在需确认配置下会先返回 `confirmation`，再根据后续输入执行或拒绝。

**说明**：当 `kernel_config.yaml` 中 `tool_confirmation_required` 包含 `echo` 时，第一次请求会得到 `confirmation`。第二次请求须**与第一次相同的 `user_id` 与 `channel`**，否则待确认状态无法匹配。确认方式：HTTP 下用 `kind: "system_command"` + `text: "yes"` / `"no"`，或 `user_message` + `text: "yes"`；也可用 `text: "/confirm <confirmation_id> yes"`（从第一次响应里复制 `confirmation_id`）。

**步骤**（整行复制执行）：

1. 触发工具（需确认时）：
```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http","text":"/tool echo 测试内容"}'
```

2. 确认执行（与上一步同一 `user_id` / `channel`）：
```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"system_command","user_id":"u1","channel":"http","text":"yes"}'
```

**通过标准**：

- 若配置了 echo 需确认：返回 `events` 中有一条 `kind: "confirmation"`，且包含 `tool_call`（如 name 为 echo、arguments 含测试内容）。
- 若未配置确认：返回 `kind: "reply"`，且内容与 echo 结果相关（如包含“测试内容”或工具执行结果）。
- 第二步在确认后：应出现 `reply` 或 `status`（如 `confirmation: approved`）及与 echo 相关的结果，而非再次卡在 `confirmation`。

---

### 2.7 设备请求占位：/tool take_photo

**目的**：验证 take_photo 触发设备请求事件，且返回结构符合 PortEvent 约定。

**步骤**（整行复制执行）：

```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http","text":"/tool take_photo"}'
```

**通过标准**：

- 若需确认：先出现 `kind: "confirmation"`；在确认通过后（或当前实现下等价行为），出现 `kind: "device_request"`，且包含 `device_request_id` 与 `request`（如 device/command 等）。
- 若无需确认：返回中应出现 `device_request` 或相应 reply 中体现已发起设备请求。

---

### 2.8 异常与边界

**目的**：保证非法或非常规输入不会导致崩溃，且返回明确事件。

| 用例       | 通过标准 |
|------------|----------|
| 缺少 text  | 不 500；`events` 中有一条 `kind: "error"`，`reason` 为 `validation_missing_text`（不进入模型）。 |
| 错误 kind  | 若被拒绝则返回 422 或 error 类事件；若接受则按实现约定。 |
| 空 text    | 不 500；返回 reply 或 error（须显式传 `"text":""`，与「缺字段」区分）。 |

可选：用下列命令测试边界（整行复制执行）。

缺少 text：
```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http"}'
```

空 text：
```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http","text":""}'
```

---

### 2.9 归档：`/archive` 与 `GET /archives`

**目的**：验证会话可标记为 `ARCHIVED`，且在 **SQLite** 后端下写入 `session_archives` 并可通过 HTTP 列出。

**说明**：

- 默认 `runtime.yaml` 为 `session_store.backend: sqlite`（会话落盘）；归档会写入 `session_archives`，`GET /archives` 可返回非空列表。若曾改为其他配置，请确认 **`backend: sqlite`** 后重启 HTTP 服务。

**步骤**（`user_id` 与下列命令保持一致，示例为 `u_archive`）：

1. 先建立至少一轮对话：
```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u_archive","channel":"http","text":"归档测试前的一句话"}'
```

2. 发送归档指令：
```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u_archive","channel":"http","text":"/archive"}'
```

3. 查询该用户的归档摘要列表：
```
Invoke-RestMethod -Uri "http://127.0.0.1:8000/archives?user_id=u_archive&limit=10"
```

**通过标准**：

- 第 2 步：`events` 中含 `kind: "reply"`，正文含「归档」相关提示（如「会话已归档」）。
- 第 3 步：HTTP 200；JSON 含 `ok: true`、`archives` 数组、`version`（或与当前实现一致字段）。
- **`archives` 至少一条**（在已执行 `/archive` 且会话有对话的前提下），条目含 `session_id`、`summary_text`、`archived_at` 等（与实现一致）。

---

### 2.10 MCP 演示工具（可选，`/tool ping` / `/tool add`）

**目的**：在启用 `mcp_servers.yaml` 且 `kernel_config.yaml` 白名单含 `ping`、`add` 时，验证经 MCP stdio 调用演示 Server。

**前置**：`src/platform_layer/resources/config/mcp_servers.yaml` 中 **`enabled: true`**（仓库默认已为 `true`；若你改为 `false` 需改回），重启服务；已 `pip install` 主 `requirements.txt`（含 `mcp`）。

**步骤**（整行复制）：

1. ping：
```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http","text":"/tool ping"}'
```

2. add：
```
Invoke-RestMethod -Method Post -Uri "http://127.0.0.1:8000/input" -ContentType "application/json; charset=utf-8" -Body '{"kind":"user_message","user_id":"u1","channel":"http","text":"/tool add 2 3"}'
```

**通过标准**：

- `events` 中出现与工具执行相关的 `reply` 或 `status`，且 MCP 未报错时，正文或结构化结果中可体现 `pong` 或与 `add` 一致的结果（如 `5`）；若 MCP 未启用或进程失败，允许出现含 `mcp_error` 的说明，不视为本用例必失败（仅作可选回归）。

## 四、通过标准汇总

- **健康检查**：/health 返回 `ok: true`（并含 `version`，与 `src/app/version.py` 一致）。
- **普通对话**：/input 单轮 user_message 得到 reply 且非“未配置”类错误。
- **上下文维持**：同一 user_id 下两句对话，第二句能引用第一句内容。
- **/help、/summary**：返回本地约定文案，非模型自由发挥。
- **归档**：`/archive` 返回约定 reply；若使用 **SQLite**，`GET /archives?user_id=` 能列出对应用户的归档摘要（见 2.9）。
- **MCP（可选）**：见 2.10；依赖 `mcp_servers.yaml` 与 `mcp` 包。
- **/tool echo、/tool take_photo**：按配置出现 confirmation 或 device_request，且结构符合 PortEvent 定义。
- **异常与边界**：无 500，返回中为 reply/error/status 等已知事件类型。

满足以上即可视为当前版本系统测试通过。若某用例依赖尚未实现（如 HTTP 下确认二次输入），可在文档中注明“仅 CLI 验证”或“待实现”，避免误判为失败。**归档列表非空**依赖默认 **`runtime.yaml` 为 `backend: sqlite`**（见 2.9）；若曾改为非法/旧配置，请恢复后重启服务。
