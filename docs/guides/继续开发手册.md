## 2. 产品价值与边界（共识）

- **价值**：宿主主要通过 **YAML + 可选 setuptools entry_points** 装配 **模型、工具、存储、提示词、安全策略参数**，复用统一 **会话 + loop + 确认/设备流 + 记忆编排**。
- **边界**：
  - **声明式替换**默认含义为 **改配置/装插件后重启进程**重新 `build_core`，而非进程内无重启热切换。
  - **多 worker**：待确认/待设备状态当前在 **Port 进程内**；水平扩展需宿主外置状态或粘性会话（见开发状态文档）。

## 4. 主装配链路与配置地图

自上而下（与 `app/composition.build_core` 一致）：

```
platform_layer/resources/config/
  resource_index.yaml      → active_security_policy、active_storage_profile、active_resource_access_profile
  security_policies.yaml   → 守卫、风险、限流等
  storage_profiles.yaml    → 会话/归档路径、archive_store_ref、memory.path（记忆库路径）
  runtime.yaml             → session_store 路径；**port.interaction_mode_ref**（CLI InteractionMode）
  kernel_config.yaml       → 全局 loop、工具白名单/确认、关卡② `context_isolation_enabled`
  session_defaults.yaml    → ConfigProvider 默认会话（model、skills、limits…）
  model_providers.yaml     → providers、backend、model_backend_ref…
  prompts.yaml             → 按 provider 合并进 registry
  skills.yaml              → 技能条目
  tools.yaml               → local_handlers、device_routes、entrypoints
  mcp_servers.yaml         → MCP 开关、`bridge_ref`、stdio servers（可选）
  memory_policy.yaml       → 记忆开关、dual_store_ref、embedding_ref、可选 embedding_openai、检索参数…
```

**校验入口**：`app/config_loaders/resource_validation.validate_resource_configs`（扩展新 YAML 字段时通常需同步更新校验逻辑）。

## 5. Entry points 速查表

| 配置 ref 前缀 | Entry point group | 工厂/可调用形态（以各 `resolve_*`  docstring 为准） |
|---------------|-------------------|-----------------------------------------------------|
| `entrypoint:<name>` 模型后端 | `pompeii_agent.model_backends` | Chat 后端可调用 |
| `kernel.prompt_strategy_ref` / `params.prompt_strategy_ref` 的 `entrypoint:<name>` | `pompeii_agent.prompt_strategies` | `(*, system_prompt, provider, session, context, skill_registry) -> str \| None` |
| `entrypoint:<name>` 工具 | `tools.yaml` 的 `entrypoint_group`（默认 `pompeii_agent.tools`） | 与 `local_handlers` 合并 |
| `entrypoint:<name>` 会话存储 | `pompeii_agent.session_stores` | `factory(sqlite_path: Path) -> SessionStore` |
| `entrypoint:<name>` 双库记忆 | `pompeii_agent.memory_dual_stores` | `factory(memory_db_path: Path) -> DualMemoryStore` |
| `entrypoint:<name>` 嵌入 | `pompeii_agent.embedding_providers` | `factory(embedding_dim: int, policy: MemoryPolicyConfig \| None) -> EmbeddingProvider` |
| `entrypoint:<name>` 守卫评估 | `pompeii_agent.guard_evaluators` | `(text: str) -> bool` 等 |
| `entrypoint:<name>` 守卫模型 | `pompeii_agent.guard_models` | `(text: str) -> bool`（应拦截为 True） |
| `entrypoint:<name>` 旧长期记忆库 | `pompeii_agent.long_term_memory_stores` | **registry 存在，主装配未用** |
| `entrypoint:<name>` MCP 工具桥 | `pompeii_agent.mcp_bridges` | `factory(cfg: McpRuntimeConfig, src_root: Path) -> McpToolBridge \| None` |
| `skills.enable_entrypoints` + `entrypoint_group` | `pompeii_agent.skills`（默认） | 可调用 `() -> SkillSpec \| dict`，或返回 `SkillSpec`；**name = skill id** |
| `kernel.tool_policy_engine_ref` 的 `entrypoint:<name>` | `pompeii_agent.tool_policies` | `(*, tool_call, kernel_config, bypass_tool_confirmation) -> ToolPolicyDecision` |
| `kernel.loop_policy_engine_ref` 的 `entrypoint:<name>` | `pompeii_agent.loop_policies` | `(session, kernel_config) -> LoopGovernance` |
| `runtime.port.interaction_mode_ref` 的 `entrypoint:<name>` | `pompeii_agent.interaction_modes` | `() -> InteractionMode`（CLI stdin 循环用） |

插件包需在 `pyproject.toml` / `setup.cfg` 中声明对应 group 的 entry point，安装后重启内核进程。

## 6. 已知缺口与推荐优先级（Backlog）

### ver0.4 分阶段里程碑（全景，非一次迭代做完）

| 阶段 | 内容 | 状态（快照） |
|------|------|----------------|
| M1 | 双库 SQLite + Orchestrator + hash 嵌入 + FTS/RRF + `/remember`/`/forget`/归档晋升 | 已落地 |
| M2 | 双库/嵌入 **registry** + `dual_store_ref` / `embedding_ref` | 已落地 |
| M3 | **OpenAI 兼容嵌入** `builtin:openai_compatible` + `embedding_openai` 配置块 | 已落地（0.4.25） |
| M4 | 独立向量服务 / HNSW、GraphRAG、多模态组装 | 未落地 |
| M5 | Workflow、用户画像、关卡⑤ 扩展（通用资源目录/审批流） | **关卡⑤ MVP**（`resource_access` + 长期记忆）已落地；其余未落地；**LLM 归档摘要**已部分落地 |

### P0 — 闭合「双故事线」与宿主预期

1. **`LongTermMemoryStore` vs `DualMemoryStore`** — **已封顶（0.4.29）**  
   - 主线：`MemoryOrchestrator` + `dual_store_ref`；旧线不接入 `composition`。  
   - **约束**：`memory_policy.enabled=true` 时 `storage_profiles.memory.store_ref` **必须** `builtin:noop`，避免与同路径双库冲突（`resource_validation`）。  
   - 旧 registry 保留作插件/测试；若未来要「双线写入」须另选路径与明确设计，不在当前默认内。

### P1 — 「可分发库」宿主体验

1. **可安装包**：补全 `pyproject.toml` 的 `[project]`、包路径与可选 `console_scripts`。  
2. **程序化配置门面**：例如从 `dict`/单文件构造与 YAML 等价对象，减少强制拷贝整棵 `platform_layer`。  
3. **最小示例**：`examples/minimal_kernel.py`（`build_core` + 一次 `handle`）。

### P2 — 对称插拔（按需）

1. **MCP 桥**：**已实现** `bridge_ref` + `pompeii_agent.mcp_bridges`；宿主可换 stdio 以外实现（仍须符合 `McpToolBridge`）。  
2. **AssemblyModule**：是否允许 `entrypoint:<name>` 或 `build_core(assembly=...)` 显式注入（已有参数扩展空间时需统一文档）。  
3. **Skill 加载**：已支持可选 entrypoint（`skills.yaml` + `pompeii_agent.skills`）；纯 YAML 仍为默认。

### P3 — 流程与提示词「引擎级」扩展

1. **Loop/tool policy**：若需无 fork 替换，引入小型 `policy_backend_ref`（设计成本较高，谨慎）。  
2. **提示词后处理**：**MVP** 已支持 `prompt_strategy_ref` + `prompt_strategies` entrypoint；更复杂的「多段模板引擎」仍可后续扩展。

## 7. 开发约束（与仓库规则对齐）

- **分层**：`core` 不依赖 `app`/`port`；新能力优先通过 **Protocol + composition 注入**。  
- **STUB**：占位实现须带 `STUB(YYYY-MM-DD): …`（见 `design/ai-rules-template/RULES.md`）。  
- **变更记录**：功能/行为变更须追加 `design/CHANGELOG.md`。  
- **测试**：新 registry 须有 `tests/test_*_registry.py` 或集成测试；改校验须覆盖非法配置用例。
